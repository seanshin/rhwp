## 1. Task #265 — HWP 3.0 파일 감지 + 친절한 에러 메시지

### 배경

@jangster77 님의 제보 (2026-04-23). `samples/issue_265.hwp` 를 rhwp 의 어떤 경로에서도 열지 못함. 원인: **HWP 3.0** 바이너리 포맷 (1996~1998년 경 한글 3.0). 현재 rhwp 는 HWP 5.0 + HWPX 만 지원.

### 처리 방침
작업지시자 지시: "HWP 3.0 정식 지원은 이번 마일스톤 범위 아님. 감지 + 친절한 에러 메시지까지만".

### 변경 범위
- **코드** `src/parser/mod.rs`:
  - `FileFormat::Hwp3` variant + `detect_format` 에 17바이트 프리픽스 체크
  - `ParseError::UnsupportedFormat { format, hint }` + `Display` 확장
  - `parse_document` 에서 Hwp3 → 한글 안내 메시지 반환
- **코드** `src/error.rs` — **발견된 추가 버그 수정**:
  - `From<ParseError> for HwpError` 등 3개 `From` 이 `{:?}` (Debug) 를 쓰고 있어 친절한 Display 메시지가 사용자에게 도달하지 못함. `{e}` (Display) 로 전환. CFB 매직 바이트 내부 누출도 함께 해결
- **코드** `rhwp-studio/src/main.ts` — **UI 전달 누락 수정**:
  - 기존: 상태 표시줄 (22px, `overflow: hidden`) 에만 쓰고 있어 긴 메시지 대부분 가려짐
  - 수정: `showLoadError` 유틸 + 우상단 토스트 (`showToast` 기존 유틸 재사용). 3 로드 경로 (`loadFile` · `loadFromUrlParam` · `open-document-bytes` 이벤트) 통합
  - `open-document-bytes` 이벤트 핸들러가 WASM throw 를 catch 하지 않아 "파일 열기" 커맨드 에선 에러가 완전 묵살되던 문제도 함께 수정
- **테스트** 6건 (detect + parse + Display 회귀 방어)
- **WASM 재빌드**: 4,076,166 → 4,081,046 bytes (+4,880 bytes)

## 2. Task #294 — 목차 교차-run 오른쪽 탭 right edge 정렬

**이슈**: [#294](https://github.com/edwardkim/rhwp/issues/294)  
**마일스톤**: M100 (v1.0.0)  
**브랜치**: `local/task294`  
**커밋**: `71b6a73` (수정) · `549d41a` (문서)

### 핵심 발견 — 장제목 vs 소제목 run 구조 차이

| 유형 | run 구조 |
|------|---------|
| 장제목 | `[제목]` `["\t"]` `[" "]` `["3"]` — tab 단독 run |
| 소제목 | `[번호]` `["제목\t "]` `["4"]` — tab이 제목 run 끝에 후행 공백과 포함 |

소제목 run은 `ends_with('\t')` = false → 교차-run 탐지 미발화 → 페이지번호가 tab_pos_abs에서 시작 → right edge 초과.

### 수정 2건

**수정 1** — look-ahead 총 폭 합산 (tab_type=1):

```rust
// 수정 전: x = col_area.x + tab_pos - leading_ws_w;
// 수정 후:
let mut total_w = next_w;
for fr in &comp_line.runs[(run_idx + 1)..] {
    let mut fts = resolved_to_text_style(styles, fr.char_style_id, fr.lang_index);
    fts.default_tab_width = tab_width;
    fts.tab_stops = tab_stops.clone();
    fts.auto_tab_right = auto_tab_right;
    fts.available_width = available_width;
    total_w += estimate_text_width(&fr.text, &fts);
}
x = col_area.x + tab_pos - total_w;
```

**수정 2** — 교차-run 탐지 조건 개선:

```rust
// 수정 전:
if has_tabs && run.text.ends_with('\t') {
// 수정 후 (후행 공백만 제거, 탭 보존):
if has_tabs && run.text.trim_end_matches(' ').ends_with('\t') {
```

`trim_end()` 는 `"\t"` 단독 run을 `""` 로 만들어 장제목 탐지 실패 → `trim_end_matches(' ')` 사용.

### 검증 결과

| 페이지번호 | 수정 전 | 수정 후 | 목표 | 오차 |
|-----------|--------|--------|------|------|
| 단일 '3','8' (장제목) | 709.09 | 700.09 | 699.76 | +0.33px ✓ |
| 단일 '4'~'9' (소제목) | 709.21 | 700.09 | 699.76 | +0.33px ✓ |
| 두 자리 소제목 | 718.54 | 699.42 | 699.76 | -0.34px ✓ |
| 두 자리 장제목 | 718.76 | 699.76 | 699.76 | -0.00px ✓ |

### 교훈

`trim_end()` 와 `trim_end_matches(' ')` 의 차이: `trim_end()` 는 탭·개행 포함 모든 whitespace 제거, `trim_end_matches(' ')` 는 공백만 제거. 탭 검출 로직에서 반드시 구분해야 함.
